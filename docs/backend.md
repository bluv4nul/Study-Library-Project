## Описание выполненной части

В рамках проекта "Электронная библиотека" была реализована серверная часть приложения, отвечающая за CRUD-операции и REST API. Серверная часть предоставляет единый программный интерфейс для работы с книгами, авторами, жанрами и издательствами, а также обеспечивает сохранение изменений в базе данных MongoDB.

Основная задача backend-части - связать пользовательский интерфейс и базу данных. Клиентское приложение может отправлять HTTP-запросы на сервер, а сервер выполняет нужные операции с коллекциями MongoDB и возвращает результат в формате JSON.

Реализация находится в папке `backend` и написана на Python с использованием FastAPI и PyMongo.

## Используемые технологии

Для реализации серверной части были использованы следующие технологии:

| Технология | Назначение |
| ---------- | ---------- |
| Python | Основной язык серверной части |
| FastAPI | Создание REST API и автоматической Swagger-документации |
| PyMongo | Подключение к MongoDB и выполнение запросов к базе данных |
| Uvicorn | ASGI-сервер для запуска FastAPI-приложения |
| Docker | Контейнеризация backend-сервиса |
| Docker Compose | Совместный запуск backend и MongoDB |

## Структура backend-части

Основные файлы серверной части:

```text
backend/
  app/
    __init__.py
    main.py
  Dockerfile
  requirements.txt
  .dockerignore
```

Файл `backend/app/main.py` содержит основную логику приложения:

* подключение к MongoDB;
* описание FastAPI-приложения;
* настройку CORS;
* маршруты для CRUD-операций;
* фильтрацию, сортировку и пагинацию;
* преобразование `ObjectId` в строки для JSON-ответов;
* проверку существования связанных документов;
* обработку ошибок.

Файл `requirements.txt` содержит Python-зависимости проекта, а `Dockerfile` описывает сборку контейнера backend-сервиса.

## Подключение к базе данных

Сервер подключается к MongoDB через переменные окружения:

```text
MONGO_URI=mongodb://database:27017
DB_NAME=electrolibrary
```

В Docker Compose backend обращается к MongoDB по имени сервиса `database`, поэтому внутри контейнерной сети используется адрес:

```text
mongodb://database:27017
```

Если сервер запускается локально без Docker, по умолчанию используется:

```text
mongodb://localhost:27017
```

Работа выполняется с базой данных `electrolibrary` и коллекциями:

* `books`;
* `authors`;
* `genres`;
* `publishers`.

## REST API

REST API построен вокруг основных сущностей предметной области: книг, авторов, жанров и издательств. Все маршруты начинаются с префикса `/api`.

Для проверки работоспособности сервера реализован служебный маршрут:

| Метод | Маршрут | Назначение |
| ----- | ------- | ---------- |
| GET | `/api/health` | Проверка состояния API |

Пример ответа:

```json
{
  "status": "ok",
  "database": "electrolibrary"
}
```

## CRUD для книг

Книги являются основной сущностью приложения. Для них реализован полный набор CRUD-операций.

| Метод | Маршрут | Назначение |
| ----- | ------- | ---------- |
| GET | `/api/books` | Получить список книг |
| GET | `/api/books/{book_id}` | Получить одну книгу по идентификатору |
| POST | `/api/books` | Создать новую книгу |
| PATCH | `/api/books/{book_id}` | Частично обновить книгу |
| PUT | `/api/books/{book_id}` | Полностью заменить книгу |
| DELETE | `/api/books/{book_id}` | Удалить книгу |

При получении книги сервер возвращает не только поля самой книги, но и связанные документы:

* авторов из коллекции `authors`;
* жанры из коллекции `genres`;
* издательство из коллекции `publishers`.

Для этого используется агрегация MongoDB с операциями `$lookup` и `$unwind`.

Пример структуры ответа для книги:

```json
{
  "_id": "OBJECT_ID",
  "title": "1984",
  "year": 1949,
  "isbn": "978-0-452-28423-4",
  "author_ids": ["AUTHOR_ID"],
  "genre_ids": ["GENRE_ID"],
  "publisher_id": "PUBLISHER_ID",
  "authors": [
    {
      "_id": "AUTHOR_ID",
      "author_name": "George Orwell"
    }
  ],
  "genres": [
    {
      "_id": "GENRE_ID",
      "genre_name": "Dystopia"
    }
  ],
  "publisher": {
    "_id": "PUBLISHER_ID",
    "publisher_name": "Penguin Books"
  }
}
```

## Создание книги

Для создания книги клиент отправляет POST-запрос на маршрут `/api/books`.

Пример JSON:

```json
{
  "title": "Название книги",
  "year": 2026,
  "isbn": "978-1-234-56789-0",
  "author_ids": ["AUTHOR_ID"],
  "genre_ids": ["GENRE_ID"],
  "publisher_id": "PUBLISHER_ID"
}
```

Перед сохранением сервер выполняет проверки:

* поле `title` должно быть передано;
* поле `year` должно быть целым положительным числом;
* поле `isbn` должно быть передано;
* `author_ids` должен быть массивом идентификаторов;
* `genre_ids` должен быть массивом идентификаторов;
* `publisher_id` должен быть корректным `ObjectId`;
* связанные авторы, жанры и издательство должны существовать в базе данных.

Если данные корректны, новая книга сохраняется в коллекцию `books`, а сервер возвращает созданный документ.

## Редактирование книги

Для редактирования книги реализованы два метода:

| Метод | Особенность |
| ----- | ----------- |
| PATCH | Меняет только переданные поля |
| PUT | Полностью заменяет документ книги |

Пример частичного обновления:

```json
{
  "title": "Новое название",
  "year": 2027
}
```

Если книга с указанным идентификатором не найдена, сервер возвращает ошибку `404`.

## Удаление книги

Удаление выполняется запросом:

```text
DELETE /api/books/{book_id}
```

Если книга существует, она удаляется из коллекции `books`, а сервер возвращает статус `204 No Content`. Если книги с таким идентификатором нет, возвращается ошибка `404`.

## Фильтрация, сортировка и пагинация книг

Для списка книг реализованы параметры запроса, которые позволяют искать, фильтровать и сортировать данные.

| Параметр | Назначение |
| -------- | ---------- |
| `search` | Поиск по названию книги |
| `year` | Фильтр по точному году издания |
| `yearFrom` | Минимальный год издания |
| `yearTo` | Максимальный год издания |
| `publisher_id` | Фильтр по издательству |
| `author_id` | Фильтр по автору |
| `genre_id` | Фильтр по жанру |
| `sort` | Поле сортировки |
| `order` | Направление сортировки: `asc` или `desc` |
| `page` | Номер страницы |
| `limit` | Количество элементов на странице |

Примеры запросов:

```text
GET /api/books?search=1984
GET /api/books?year=1949
GET /api/books?yearFrom=1930&yearTo=1950
GET /api/books?author_id=OBJECT_ID
GET /api/books?genre_id=OBJECT_ID
GET /api/books?publisher_id=OBJECT_ID
GET /api/books?sort=year&order=desc&page=1&limit=10
```

Ответ списка содержит массив данных и метаинформацию для пагинации:

```json
{
  "data": [],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 5,
    "pages": 1
  }
}
```

## Обработка ошибок

Сервер возвращает понятные HTTP-статусы:

| Статус | Ситуация |
| ------ | -------- |
| 400 | Некорректные входные данные |
| 404 | Документ не найден |
| 409 | Нарушение уникальности, например повторяющийся ISBN |
| 500 | Внутренняя ошибка сервера |

Перед выполнением операций сервер проверяет корректность `ObjectId`. Если идентификатор имеет неправильный формат, возвращается ошибка `400`.

Также при создании и обновлении книги проверяется, что указанные авторы, жанры и издательство действительно существуют в базе данных. Это позволяет поддерживать логическую целостность связей на уровне приложения.

## Запуск через Docker

Backend запускается через Docker Compose вместе с MongoDB. В `docker-compose.yaml` добавлен сервис `api`, который собирается из папки `backend`.

Фрагмент конфигурации:

```yaml
api:
  build:
    context: ./backend
  environment:
    PORT: 3000
    MONGO_URI: mongodb://database:27017
    DB_NAME: electrolibrary
  ports:
    - 3000:3000
  depends_on:
    - database
```

Команда запуска проекта:

```bash
docker compose up --build
```

После запуска API доступен по адресу:

```text
http://localhost:3000
```
